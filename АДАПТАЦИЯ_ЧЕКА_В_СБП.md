# Адаптация чека с полным набором букв в формат СБП

## Да, можно

Если у вас есть чек **в любом формате** (карта на карту, другой банк, другой тип перевода), но в нём **все нужные символы** — можно взять из него **шрифт и CMap** и подставить в чек СБП.

## Как это работает

1. **Источник (source)** — ваш чек с полным алфавитом (любой формат).
2. **Цель (target)** — оригинальный чек СБП (layout, заголовок «Исходящий перевод СБП», поля).
3. В выходной PDF сохраняются:
   - структура, layout и тексты меток от **target** (СБП),
   - шрифт и ToUnicode от **source** (полный набор букв).

## Команда

```bash
python3 copy_font_cmap.py "ваш_чек_с_полными_буквами.pdf" "проходящий_чек_СБП.pdf" "sbp_with_full_font.pdf"
```

Потом используйте результат как шаблон:

```bash
python3 gen_verified_receipt.py out.pdf --template sbp_with_full_font.pdf --payer "Фуя Ян У." ...
```

## Важные условия

| Условие | Зачем |
|---------|-------|
| Оба чека — **VTB** (openhtmltopdf) | Совместимость CID→GID, структуры шрифта |
| Target — **проходящий** чек СБП | Layout должен быть тем, что проверяет бот |
| Source — любой тип чека ВТБ | Карта на карту, СБП, другой — не важно, берётся только шрифт |

## Когда может не сработать

- **Разные генераторы PDF**: source из Альфы (Oracle BI Publisher), target из ВТБ (openhtmltopdf) — могут не совпадать CID→GID, часть символов отобразится неверно.
- **Разные банки**: шрифты и CMap могут быть устроены по-разному.

## Рекомендуемая последовательность

1. Скиньте чек с полным набором букв.
2. Я проверю, что он из того же семейства (openhtmltopdf, ВТБ).
3. Запустим `copy_font_cmap` с вашим чеком как source и проходящим СБП как target.
4. Получим `sbp_with_full_font.pdf` — СБП с полным алфавитом.
5. Используем его как шаблон для генерации любых ФИО.
