# Почему бот может не принять чек

## 1. Имя — некорректные буквы (исправлено)

**Причина:** Буква **А** (кириллица) отсутствовала в CMap donor'а. Добавляли её через Identity (`CID = Unicode`), но шрифт использует свои CIDs, и глиф при CID 0410 отображался неверно.

**Решение:** Сначала использовать homoglyph (А→A, латиница). A есть в CMap, выглядит так же. ToUnicode не трогаем.

## 2. Общие причины отказа бота

- **Модификация ToUnicode** — добавление записей в CMap меняет структуру PDF. Если бот проверяет хеш или эталонные CMap, чек может не пройти.
- **Формат operation_id** — C260307004747581 может не соответствовать внутреннему формату банка.
- **Document ID** — подмена на случайный: бот мог сравнивать с базой известных ID.
- **account** — замена счёта часто вызывает срабатывание (см. АНАЛИЗ_СЧЁТ_CID.md).

## 3. Рекомендации

- Использовать donor'а, в котором уже есть все нужные символы.
- Не менять номер счёта.
- Оставить Document ID donor'а (`keep_donor_id: true`), если бот сверяет его.
- Использовать реальный operation_id из настоящего чека, если бот проверяет его формат/валидность.
