# Почему чек пересобран и размер большой

## Как устроена генерация

1. **CID-патч** (~59 KB) — точечная замена текста в PDF, структура и размер почти не меняются.
2. **Rebuild** (~68 KB и больше) — полная пересборка через `rebuild_pdf.py`, размер увеличивается.

## Что произошло с receipt_sbp_5000.pdf

Использовался **rebuild**, потому что **CID-патч не сработал**.

### Причина

Donor `20251021220717 (1).pdf` (первый в папке СБП по алфавиту) имеет **ограниченный CMap** — в нём нет символов:

- **Р**, **г** — для «Руслан Сергеевич»
- **+**, **(**, **)**, **-** — для формата «+7 (961) 315-70-70»

`find_best_donor` по всем PDF в СБП не находит ни один с нужным набором символов (нет PDF, где есть и Р, и г, и форматирование телефона с скобками и плюсом).

Из-за этого:
- берётся donor `20251021220717 (1).pdf`;
- CID-патч падает (невозможно закодировать новый текст);
- включается **rebuild** → размер ~68 KB вместо ~59 KB.

## Что можно сделать

### 1. Формат телефона без скобок (рекомендуется)

Использовать номер **без форматирования**: `79613157070` вместо `+7 (961) 315-70-70`.

Такой формат укладывается в CMap большинства donor'ов → CID-патч срабатывает → чек ~58 KB вместо ~68 KB.

**receipt_sbp_5000.pdf** перегенерирован с телефоном `79613157070` — размер 58 KB.

### 2. Ручной выбор donor

Явно указать donor, у которого в CMap есть все нужные символы:

```bash
python3 generate_receipt.py --json receipt.json --donor document06.03.26 01_53_12.520.pdf
```

(если именно этот donor покрывает все символы для твоих данных)

### 3. Donor’ы как шаблоны

Чеки всегда собираются на основе **готового donor-PDF**. Отдельного «с нуля» создания PDF нет — только замена нужных полей в выбранном donor.
