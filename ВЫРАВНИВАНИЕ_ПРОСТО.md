# Выравнивание правого столбца — простая модель

## Суть

**Один параметр: wall** — X-координата, где должен заканчиваться последний символ каждого поля.

Всё. Никаких pts по полям, калибровок, layout_overrides.

---

## Как это работает

В PDF мы задаём **начало** текста (tm_x), а не конец. Поэтому:

```
tm_x = wall - ширина_текста
```

Ширина текста = сумма (ширина каждого глифа + керн между ними). Она определяется шрифтом и керном в TJ. Мы **не храним** её в конфиге — берём из того же блока донора, который заменяем:

```
ширина_на_глиф = (wall - tm_x_старого_блока) / n_глифов
```

То есть: донор уже выровнен по wall. У найденного блока (tm_x, n) → `pts = (wall - tm_x) / n`. Этим pts рисуем новый текст: `new_tm_x = wall - n_new * pts`.

Fallback: если pts получился нелепый (≤0 или >15), берём разумное среднее по керну:
- `-16.66667` (дата, ФИО, телефон, банк) → ~4.6
- `-11.11111` (сумма) → ~6.75
- `-21.42857` (получатель по центру) → ~4.2

---

## Что убрать

- `layout_config.json` pts — не нужны при замене по Y (pts из блока)
- `layout_overrides.json` pts — только для редкого fallback, можно одним значением
- `scan_donors_align.py` pts — только wall и Y полезны
- CALIBRATION.md итерации по pts — избыточны, если всегда используем pts из блока

---

## Итого

| Нужно | Не нужно |
|-------|----------|
| wall (из «Выполнено») | pts в конфиге |
| Y для сопоставления полей | scan_donors для pts |
| pts = (wall - tm_x)/n из блока | Калибровка pts по полям |
