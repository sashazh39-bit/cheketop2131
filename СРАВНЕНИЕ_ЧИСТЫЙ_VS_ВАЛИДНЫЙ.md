## Почему бот может не принимать

### 1. Document ID (мы его меняем)

Мы вызываем `patch_unique_id()` и заменяем ID на случайный. Бот может:
- хранить whitelist известных Document ID;
- считать чек валидным только если его ID есть в базе.

**Попытка:** не менять Document ID при генерации «чистого» чека — оставить оригинальный от донора.

**Реализовано:** в `receipt_sbp_clean.json` включено `"keep_donor_id": true`. Либо флаг `--keep-donor-id` в CLI.

### 2. API проверка согласованности (StopScam / НСПК)

В `АНАЛИЗ_ЧЕК_НЕ_РАСПОЗНАН.md` описано: бот может отправлять в API пару (operation_id, sbp_id, amount, account).

- В AM_1772658254522: `C160403262149539`, `A60631903485960P...`, сумма **500 RUR**, счёт **02476**.
- У нас: те же operation_id и sbp_id, но сумма **15 000 RUR** и счёт **09948**.

API может вернуть, что для `C160403262149539` ожидаются сумма 500 и счёт 02476 → несоответствие → «чек не распознан».

**Вывод:** нужны данные из одной реальной операции. Нельзя смешивать operation_id/sbp_id от одной операции с amount/account от другой.

### 3. Хеш файла

Любое изменение содержимого меняет хеш. Если бот сверяет хеш с базой, пересобранный PDF всегда будет отличаться.

## Что менять, если меняешь номер счёта

1. **Не трогать Document ID** — оставить ID от donor (AM_1772658254522). Добавлена опция `--keep-donor-id`.

2. **Использовать данные одной операции** — operation_id, sbp_id, amount, account, date_time должны быть из одного реального чека. Если меняешь счёт на 09948, нужен чека с 09948 и его реальными operation_id и sbp_id.

3. **Менять только то, что реально допускается** — возможно, бот принимает изменения счёта только в пределах одной операции (например, маскировка). Менять счёт с 02476 на другой (09948) может быть запрещено логикой проверки.
