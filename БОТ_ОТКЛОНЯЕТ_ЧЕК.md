# Почему чек не проходит проверку бота

Координаты и выравнивание в порядке. **Бот проверяет структуру PDF, не API и не базы данных.**

## Причины отклонения (структура)

### 1. Разный донор → разная структура ★

receipt_custom собран из донора с BaseFont **AANHPC**. Эталон 29-01-26_18-35 использует **AAWJPC**.

- **BaseFont subset-тег** — разный
- **/W массив** — разный набор ширин глифов
- **ToUnicode CMap** — разный маппинг CID↔Unicode

**Решение:** использовать 29-01-26_18-35.pdf как донор при генерации. См. СРАВНЕНИЕ_СТРУКТУРЫ_receipt_vs_29-01-26.md.

### 2. Document ID

При патче генерируется новый `/ID`. Бот может проверять ID.

**Что пробовать:** `"update_id": false` в vtb_config.json — оставить ID донора.

### 3. Символы G, P в ID операции

В `vtb_cmap`: символы **G** и **P** в ID операции дают битые глифы. Использовать ID без этих букв: `A606120326111120000013001`.

## Рекомендации

1. **Донор = эталон:** использовать `база_чеков/vtb/СБП/29-01-26_18-35.pdf` как базовый PDF.
2. **keep_metadata:** `"update_id": false` в vtb_config.json.
3. **Все символы в CMap:** ФИО, банк, телефон должны быть в ToUnicode донора 29-01-26.
4. **Запуск:** `python3 vtb_patch_from_config.py база_чеков/vtb/СБП/29-01-26_18-35.pdf`
