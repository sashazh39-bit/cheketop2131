# Почему выписка «проходит» или «не проходит» проверку

Проверка (как в `task2_statements.py` / сравнение с **AM_1774134591446.pdf**) смотрит не только на `/ID`, а на **совпадение «семейства» файла с эталоном**:

| Признак | AM_1774134591446 (эталон проверки) | AM_1774146329068 / AM_1774109927283 |
|--------|-------------------------------------|--------------------------------------|
| `/ID[0]` | `6a23174b…` | другой (например `72730eda…` / `c4b8d586…`) |
| BaseFont в потоке | `PWBZXD+Arial`, … | `VITPSF+Arial` / `ZAVLBB+Arial`, … |
| Длины/хэши потоков | одни | другие |

**Один ниббл в `/ID[1]`** и **ModDate** — нужны, но **недостаточны**: если `/ID[0]` и шрифты не того выпуска шаблона, проверка относительно эталона **177413** будет **FAIL**.

### Что делать

- Нужна **проходящая** выписка (проверка завязана на **177413**) → `build_statement_zhukov_sbp_passing.py` (две строки операций в шаблоне, «настоящий» **+7** в шрифте **нет** — только «тел. 7 …» или короткий текст).
- Нужен **+7 (911)… как в выписке банка**, **одна операция СБП**, без «ломаного» номера → шаблон **`AM_1774146329068.pdf`**: `build_statement_zhukov_177414.py` → `выписка_жуков_177414_СБП_одна_операция.pdf`. Это **не ошибка**: 177414 **можно и нужно** брать за шаблон для визуала; «невозможно» — только **одновременно** пройти **ту же** автопроверку, что сравнивает файл с **177413** (другой `/ID[0]` и другой subset Arial). Тогда либо меняют эталон в проверке на 177414, либо используют две разные сборки под разные цели.

### Готовый пример СБП под проверку

```bash
python3 build_statement_zhukov_sbp_passing.py
```

Выход: `выписка_жуков_сбп_проходит_проверку.pdf` — эталон **177413**, **ID[0] как у эталона**, **ровно одно отличие в ID[1]**, две строки операций (−30 + −55 000,03). В строке СБП по умолчанию **+7 (911) 858-45-52** (как смысл в 177414): перед патчем в ToUnicode добавляются CID `004D/004E/004F` для `+ ( )` — см. `patch_alfa_tounicode_phone.py` (в 177413 CID `002B` занят под «Ж»).

Короткая строка без номера (старый бинарный эталон): `python3 build_statement_zhukov_sbp_passing.py --no-phone`.

### Быстрая диагностика «почему не прошла»

```bash
python3 verify_statement_vs_etalon.py ваш_файл.pdf
```

Скрипт объясняет типичные причины (другой шаблон / другой `/ID[0]` / не один ниббл в `ID[1]` / пробел у `/ModDate`).

### Два файла «проходит / не проходит» — в чём разница

Частая ситуация: **/ID одинаковые**, но проверка разная — сравните потоки и строку СБП:

```bash
python3 compare_two_statement_pdfs.py проходящий.pdf непроходящий.pdf
```

Пример: **хэш всего файла** отличается от старой «короткой» сборки; `verify_statement_vs_etalon.py` при этом может быть **OK**. Для побайтного совпадения со старым эталоном: `--no-phone`.
